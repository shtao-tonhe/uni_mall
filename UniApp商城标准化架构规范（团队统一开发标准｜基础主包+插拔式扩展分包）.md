# UniApp商城标准化架构规范（团队统一开发标准｜基础主包\+插拔式扩展分包）

## 1\. 架构整体设计思想

本项目为**单商城业务专属架构**，不混入点餐、预约、多门店等其他行业代码。

核心设计理念：

- **基础电商 = 主包（固定、必装、核心流程）**

- **所有营销/增值功能 = 独立分包（可插拔、单独售卖、按需开启）**

- **未购买的功能，编译阶段直接剔除，不占用主包体积**

- 严格遵守微信小程序限制：**主包 ≤ 2MB**

适配商业模式：**基础版商城 \+ 按需选购扩展功能（分销/拼团/秒杀/代理/储值等）**

## 2\. 主包 \& 分包 严格边界定义（团队强制执行）

### 2\.1 主包内容（不可拆分、不可移入分包）

主包为商城**核心生存链路**，保证所有商家必用、打开极速、流程稳定。

**主包页面清单：**

- 首页、商品分类、购物车、个人中心（基础）

- 商品详情（基础信息、规格、库存、加入购物车）

- 下单、订单列表、订单详情、基础售后

- 支付页、支付结果、收货地址、登录授权

**主包资源清单：**

- 基础通用组件、基础工具函数、基础请求

- 基础用户状态、基础商品/订单逻辑

**禁止放入主包：**

- 任何分销、拼团、秒杀、代理、储值、积分等扩展业务代码

- 扩展专属弹窗、活动标签、营销逻辑

- 大图片、大资源（全部走云端OSS）

### 2\.2 扩展分包定义（所有付费功能统一标准）

**每一个独立付费功能 = 一个独立分包**，完全隔离、互不影响、可单独开启/关闭。

现有扩展分包列表：

- sub\-distribution ｜ 分销推广

- sub\-agent ｜ 代理层级

- sub\-groupbuy ｜ 拼团

- sub\-seckill ｜ 秒杀

- sub\-jielong ｜ 社群接龙

- sub\-coupon ｜ 优惠券活动

- sub\-storage ｜ 会员储值

- sub\-point ｜ 积分商城

后续新增功能**必须新建独立分包**，禁止写入主包。

## 3\. 完整目录架构（团队统一规范）

```Plain Text
client/
├── manifest.json        # 功能总开关（条件编译控制分包是否打包）
├── pages.json           # 分包动态注册（条件编译）
├── common/              # 全局公共（仅基础能力，无业务）
│   ├── api/base         # 基础电商接口（商品/订单/购物车/支付）
│   ├── components/base  # 全局基础组件
│   ├── utils            # 工具函数、路由、权限、金额处理
│   ├── store/base       # 全局基础状态
│   ├── static/base      # 极小公共图标
├── pages/               # 【主包 - 核心基础页面】
├── subPackages/         # 【所有扩展分包目录】
│   ├── sub-distribution
│   ├── sub-seckill
│   ├── sub-groupbuy
│   ├── sub-agent
│   ├── sub-storage
│   ├── sub-point
│   └── ...
├── api/ext/             # 所有扩展功能专属接口（按文件夹隔离）
├── store/ext/           # 扩展专属状态管理
├── components/ext/      # 扩展专属组件、弹窗
└── sql-ext/             # 后端配套：每个功能独立SQL表文件

sql-ext 为后端扩展模块初始化脚本目录，非备份文件。
每一个付费扩展功能对应一张独立 SQL 建表脚本，交付时按需导入，实现功能插拔开通，保证基础业务与扩展业务数据完全隔离。

sql-ext 存放每一项付费扩展功能的独立建表 SQL 文件
不是整库备份
不是数据备份
纯粹是「功能模块数据表结构初始化脚本」
举例：
分销.sql = 存放分销佣金、上下级、提现数据表结构
拼团.sql = 存放拼团活动、团单、参团记录表结构
秒杀.sql = 秒杀专属表结构

```

## 4\. 条件编译 \& 插拔式功能控制（核心业务逻辑）

### 4\.1 功能开关机制

所有扩展功能在 **manifest\.json** 配置全局开关：

- ENABLE\_DISTRIBUTION 分销开关

- ENABLE\_SECKILL 秒杀开关

- ENABLE\_GROUPBUY 拼团开关

- ENABLE\_AGENT 代理开关

**关闭状态：**打包时直接剔除该分包所有代码、页面、接口、组件。

**开启状态：**加载对应分包，展示对应入口与功能。

### 4\.2 页面入口显示规则

商品详情、个人中心首页的活动入口、营销标签：

- 通过后端返回「已开通扩展列表」动态判断显示/隐藏

- 未开通不渲染任何DOM、不加载JS、不占用体积

- 点击入口判断分包是否存在，不存在提示未开通

## 5\. 后端配套插拔规范（前后端统一）

为适配「单独购买功能、单独部署」的商业模式：

1. **基础业务：一套核心数据表**（用户、商品、分类、订单、购物车、支付）

2. **每个扩展功能：独立SQL文件、独立控制器、独立路由**

交付规则：

- 基础版：只导入基础SQL

- 增值版：按需导入 分销\.sql / 秒杀\.sql / 拼团\.sql 等

**所有扩展数据表互不耦合，不影响基础商城流程**

---

## 6\. 体积保障规范（永久杜绝2MB报错）

- 主包只保留刚需页面与最小公共资源

- 所有大图、活动图、海报全部云端OSS，本地不存

- 所有非刚需业务全部分包隔离

- 未购买功能编译直接删除，无冗余代码

- 最终主包体积永久控制在 **1\.2MB–1\.6MB**

## 8\. 底部Tab自定义配置规范（新增｜适配基础版/扩展增值版差异化）

### 8\.1 核心设计逻辑

底部Tabbar**支持后台可视化自定义配置**，仅展示「已开通基础功能、已购买扩展功能」，未开通/未购买功能禁止配置、禁止展示。完美适配不同商家套餐差异：基础款极简Tab、增值款可叠加营销功能Tab，全程不冗余、不占用主包体积。

### 8\.2 套餐默认Tab规则（强制标准）

**基础版商城（无任何扩展）默认固定Tab**：仅保留核心刚需入口，极简适配纯卖货商家

- 首页（主包核心）

- 我的（个人中心主包）

**可选自定义基础Tab（基础版可自由开启/关闭）**：均为主包页面，无体积负担

- 商品分类

- 购物车

**扩展功能Tab（仅购买对应扩展后可配置）**：全部关联分包，未开通则后台无配置选项、前端不渲染

- 分销中心、代理中心（对应分销/代理分包）

- 拼团活动、秒杀专场（对应拼团/秒杀分包）

- 积分商城、会员储值（对应积分/储值分包）

- 社群接龙、优惠券中心（对应接龙/优惠券分包）

### 8\.3 技术实现规范

1. **Tab配置数据源**：由后端返回当前商家「已开通功能列表\+自定义Tab排序\+图标\+文字」，前端无固定写死Tab配置

2. **权限拦截机制**：前端双重校验，后台配置开启 \+ 功能编译开关开启，双条件满足才渲染对应Tab，杜绝未购买功能显示

3. **分包联动规则**：扩展功能Tab被点击时，自动预加载对应分包，无卡顿；未开通功能即使被手动篡改配置，前端自动隐藏并拦截跳转

4. **体积零影响**：Tab渲染逻辑仅做判断适配，不引入任何未开通分包代码、资源，不占用主包2MB额度

### 8\.4 后台配置约束（前后端统一）

- 后台Tab配置面板，**仅展示当前商家已开通的功能模块**，无未购买扩展功能选项，从源头杜绝违规配置

- 支持自定义Tab图标、Tab文字、排序、显示/隐藏

- 基础Tab（首页、我的）为强制固定入口，不可删除，仅可调整排序
